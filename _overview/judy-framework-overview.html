RS&H CLAUDE Framework

Daunte Benjamin  ·  2026-04-29

The RS&H CLAUDE framework is a structured way to govern, distribute, and scale Claude Code across the firm. It defines where context lives, who owns what, how skills and tools are shared, and how the firm reduces redundant token spend as licenses grow.

Why this exists

RS&H has roughly thirty enterprise Claude licenses today and a long runway as adoption grows. Without structure, every user pastes the same context into every chat, skills drift, governance is informal, and token cost compounds. The framework replaces ad-hoc usage with shared context, on-demand skills, and per-domain ownership.

Cost of doing nothing. If we keep growing licenses without structure:

  • Every user re-pastes the same RS&H context into every chat. Token spend compounds.
  • Skills get reinvented per user. No shared learnings, no quality gate.
  • Memory of past decisions evaporates between people. Same questions get re-litigated.
  • Permission posture drifts. Some users on safe defaults, others bypassing prompts.
  • New hires onboard cold. Nothing inherits between sessions or between people.
  • Sensitive data leaks become a question of when, not if.

What it does

  • Educates the firm on the core primitives so users speak a common vocabulary.
  • Reduces token spend significantly by replacing pasted-in context with auto-loaded files and on-demand skills.
  • Sets governance so licenses do not fragment as the count grows.
  • Scales to seven domains using one shared scaffold. Tech proves the pattern; the others adopt it.

What this is not

Boundaries help. The framework is:

  • Not a chatbot for clients. Internal use only. Client-facing AI is a separate decision with separate guardrails.
  • Not a way to bypass IT security review. Managed settings are stricter, not looser, than ad-hoc usage.
  • Not a replacement for Power BI, SharePoint, or Microsoft Fabric. Claude reads from and works alongside those systems. It does not replace them.
  • Not HIPAA or federally regulated-data-ready today. Claude Code is not covered under Anthropic's HIPAA Enterprise plan. Federal, FDOT, CUI, ITAR data is out of scope until a separate compliance conversation happens.
  • Not a magic productivity multiplier. The gains come from discipline (shared context, governance, skills), not from the tool itself. Adoption requires the one-week pairing rotation and the weekly retro cadence.
  • Not a place for unstructured personal notes. Personal preferences live in the user-level CLAUDE.md. Org content lives in the synced framework, versioned and reviewed.

The access model

Framework files live in OneDrive and SharePoint, not in GitHub. Most RS&H users do not work in Git. Folder permissions handle access: each domain folder uses Stop Inheriting Permissions and is granted to its own SharePoint group.

Real access control, not convention. Per Microsoft documentation, OneDrive only syncs folders a user has read access to. A Finance user with access only to the org root and the Finance folder will physically never have leadership/ on their disk. Claude Code reads from the filesystem, so it can only see what synced. Other domains are invisible to the user and to Claude.

What each file actually does

Each framework file has a narrow job. Together they replace the pasted-in context, repeated explanations, and one-off setup that eat the most time and tokens today.

FileWhat it doesConcrete example
Org CLAUDE.md
RSandH-Claude/CLAUDE.md		 Firm-wide rules. Voice, security defaults, file naming, memory pointers. Auto-loads every session, anywhere in the framework. A People analyst opens Claude. The org file auto-loads. Claude already knows: no em-dashes, no PII in prompts, lowercase-hyphen filenames, RS&H voice.
Domain CLAUDE.md
<domain>/CLAUDE.md		 Adds domain-specific rules on top of the org rules. Auto-loads when the session is inside that domain. Opening Claude in technology/ adds Fabric medallion architecture conventions, snake_case SQL view rules, and the Tech vocabulary the org file does not know.
brand-voice.md The full RS&H voice spec: the four traits, the five principles, hard rules, common substitutions. When Claude drafts a client email or proposal, it pulls this in to match RS&H tone (approachably smart, genuine candor, no marketing fluff).
security-and-
confidentiality.md		 What can and cannot appear in prompts, memory, or skill outputs. PII, FDOT, CUI, ITAR, HIPAA, credentials. Escalation paths. When a prompt approaches a regulated-data boundary, Claude consults this file before continuing. The PII pre-prompt hook also reads from here.
file-naming-and-
output-standards.md		 Naming conventions, default output formats by deliverable type, SQL/Python/DAX style defaults. When Claude saves a new file, it picks the right extension and lowercase-hyphen name. SQL queries land in .sql, not pasted into a Word doc.
Templates
_org/templates/		 Starter scaffolds for new domains and new skills. Includes domain CLAUDE.md template and skill SKILL.md template. A new Domain Steward in BDO copies the domain template, fills in five placeholder sections, and is operational in five minutes.
Settings
.claude/settings.json + Intune managed		 Tool permissions, default model, hook registry. Enforced firm-wide via Intune so users cannot override. Asks before Edit, Write, or Bash outside the allowlist. Defaults to Sonnet (cheaper than Opus). Blocks --dangerously-skip-permissions.
Plugins
rsandh-org-plugin, rsandh-tech-plugin		 Bundle skills, slash commands, hooks, agents, and MCP servers into one installable package. Update once, every licensee benefits. A new Tech hire runs /plugin install rsandh-tech-plugin and inherits /postmortem, /adr, the researcher agent, and the PII scan hook.
Memory
memory/shared.md, archive/, users/		 Persistent context across sessions. Three tiers: curated shared.md (auto-loaded), quarterly archives (on-demand), personal notes (local only). The Direct Lake migration decision made last quarter lives in Tech's shared.md. Every Tech session inherits it. Nobody re-litigates the choice.
Personal CLAUDE.md
~/.claude/CLAUDE.md		 User-level preferences only: verbosity, default model, output style. Personal, not synced. A user prefers terse responses with no end-of-turn summary. They set that here. No RS&H content goes in this file; org content belongs in the synced framework.

The pattern. Three layers (Org, Domain, User) load automatically every session. Skills, scripts, and templates load on demand. Settings and hooks enforce policy without burning a Claude turn. The result is that users open Claude, type their actual question, and the framework provides the rest.

Single root folder vs. separate Teams workspaces

Two ways to physically organize the framework. The single-root model is the recommendation. The alternative, one Teams workspace per domain, doubles the admin surface and breaks the hierarchical loading that makes the framework work.

Recommended

Single root folder, seven subfolders

RSandH-Claude/
  CLAUDE.md          ← org rules
  _enterprise/       ← shared
  technology/
  finance/
  people/
  ... (7 domains)

What it gets right

  • One OneDrive folder, one sync target.
  • Hierarchical CLAUDE.md works. Org rules and domain rules layer automatically.
  • SharePoint per-folder permissions handle real access control. Other domains are invisible to users who do not belong.
  • One admin surface. Update root CLAUDE.md once, all licensees get it.
  • Cross-domain users handled by SharePoint group membership.
Costs more

Seven separate Teams workspaces

Tech-Claude/        (Teams site 1)
Finance-Claude/     (Teams site 2)
People-Claude/      (Teams site 3)
Leadership-Claude/  (Teams site 4)
... (7 sites total)

What it costs

  • Seven Teams sites to provision, govern, and maintain.
  • Hierarchical loading does not cross sites. Org rules must be copied into each site and stay in sync manually.
  • Org-wide updates fan out to seven places. Drift becomes the default state.
  • Cross-domain users juggle multiple workspaces and permissions.
  • Seven admin surfaces means more work for the Working Group every quarter.

Recommendation. Single root, seven subfolders, SharePoint permissions per folder. Same access control as separate sites, far less admin overhead, and the hierarchical CLAUDE.md model only works in this shape.

Scaling: nested sub-domains

The seven-domain structure works for the firm today. As a domain grows, sub-teams may need their own rules, memory, and stewardship. The framework supports this through nesting. Open a session in technology/data/projects/foo/ and four CLAUDE.md files auto-load (org, Tech, Data, project), each adding specificity, none repeating.

When to split a domain

Sub-team profileRecommended pattern
More than 20 people, own steward, own data classification postureNested sub-domain. Full CLAUDE.md, context, memory, skills.
10 to 20 people, shares most rules, distinct on a fewPath-scoped rules in _claude/rules/. Loads only when working in that team's files.
Fewer than 10 people, mostly shared rulesDo not split yet. Add to shared.md in the parent domain.
Almost no shared content with the parentPeer domain. Promote out of the parent into its own top-level domain folder.

Practical limit on nesting. Three to four layers (org + domain + sub-domain + project) before context loading gets noisy and contributors lose track of which rules live where. Do not nest deeper without a strong reason. If a sub-team needs its own sub-sub-domain, that is a signal it should probably promote to a peer domain.

The mental model: WAT

Claude Code is composed of three primitives that combine to do useful work. When someone asks "is that a workflow, an agent, or a tool?" they are using the WAT lens to figure out what kind of thing they are looking at.

PrimitiveWhat it isOne-line summary
WorkflowA repeatable parameterized procedureThe recipe
AgentA specialized role with isolated contextThe worker
ToolA capability Claude can callThe verb

To these, three pieces of glue and one wrapper.

GlueWhat it isOne-line summary
SkillOn-demand knowledge or workflowThe reference
HookDeterministic automation around eventsThe policy
MemoryPersistent context across sessionsThe institutional knowledge
PluginBundled, distributable packageThe container

Tool

A specific capability Claude can call. Tools are atomic. They do one thing and return a result. Tools come from three sources: built-in, Model Context Protocol (MCP) servers (extensions like a Microsoft Fabric or SharePoint integration), and custom tools built via the Agent SDK.

The built-in tools, in plain language:

ToolWhat it does
ReadOpen a file and read its contents.
WriteCreate a new file.
EditModify an existing file in place.
BashRun a shell command (for example, git status, python script.py).
GrepSearch for text patterns inside files. Like Find, but powerful and fast.
GlobFind files by name pattern (for example, all .sql files in a folder).
WebFetchFetch a web URL and return its contents.

Mental model: a verb. What Claude can do.

Agent

A specialized role with its own context window and tool access. The main agent is the session a user talks to. A subagent is invoked for a specific job, runs in a fresh context, and returns a summary. Subagents prevent stale context buildup: exploration and research do not pollute the main thread.

Mental model: a specialized worker. Who is doing the job.

Workflow

A parameterized procedure. In Claude Code, workflows show up as slash commands. A user types /adr "Postgres 16 upgrade" and Claude follows the recipe. Common RS&H Tech examples: /adr, /postmortem, /handoff, /runbook, /change.

Mental model: the recipe. How to do a specific common task.

Skill

On-demand knowledge or a workflow that Claude pulls in when relevant. Skills live in .claude/skills/<name>/SKILL.md and can include reference material, scripts, and assets. Two flavors: knowledge skills (auto-surfaced when relevant) and workflow skills (invoked explicitly or by Claude). Anthropic is consolidating around skills as the dominant abstraction.

Skill vs. tool vs. agent

  • A tool is atomic: Read a file.
  • A skill is higher level: knowledge or a recipe that uses multiple tools.
  • An agent is a sub-Claude that runs in its own context.
  • A skill might tell Claude "use this approach"; an agent is another Claude doing the work.

Hook

A script that fires automatically on a specific event. No Claude reasoning involved. Hooks are deterministic and fire one hundred percent of the time. Use a hook when "this must happen, no exceptions" (PII scanning, audit logging). Use a skill when "this is how we usually do it" (judgment is wanted).

EventFires when
PreToolUseBefore any tool call. Best for policy checks, PII redaction.
PostToolUseAfter any tool call. Best for audit logging.
OnPromptSubmitWhen a user submits a prompt. Best for prompt-level scanning.
OnStopWhen the session ends. Best for final audit log entries.

Memory and CLAUDE.md

CLAUDE.md holds always-loaded context. Claude Code reads it at session start, and the prompt cache holds it for about five minutes, so subsequent turns read it at roughly ten percent of the original input cost. Memory is persistent context across sessions. RS&H uses a three-tier memory pattern (curated shared.md, quarterly archives, per-user scratchpad) so a single file does not become unmanageable as users grow.

Plugin

A package that bundles skills, slash commands, hooks, agents, and MCP servers into one installable unit. Plugins are how the framework distributes capability. Update once; every licensee benefits. The RS&H framework ships two plugins: one cross-domain (rsandh-org-plugin) and one Tech-specific (rsandh-tech-plugin). Source lives in private GitHub; distribution flows through the Claude Enterprise marketplace.

Concrete example: how the pieces fit

A Tech user types /postmortem after an outage:

  1. The slash command (a workflow) lives in rsandh-tech-plugin/commands/postmortem.md.
  2. It activates the incident-postmortem skill in the same plugin.
  3. The skill says: first use the researcher agent to gather the timeline from logs and chat history, then walk through the postmortem template, then save the result to the Tech memory archive.
  4. The researcher agent runs in its own isolated context. It uses tools (Read, Grep, Bash) to find data and returns a 300-word summary.
  5. The main Claude takes that summary, fills the postmortem template using format conventions loaded from the Tech CLAUDE.md, and produces the final document.
  6. A hook (on-stop-audit-log.sh) fires when the session ends and writes a session entry to the audit log.

The user typed five characters. The framework did the rest.

The whole point. Tools are the hands. Agents are the brains. Workflows are the recipes. Skills are the knowledge. Hooks are the policy. Plugins are the package. CLAUDE.md is the always-loaded foundation.

Token economics

The framework reduces input-token spend significantly across steady-state work. The mechanisms:

One illustrative number. Today, roughly 30 users each paste a 500-token "RS&H is..." context blurb at the start of about 10 chats per day. That is roughly 150,000 redundant tokens per day, before anyone has asked a real question. Org CLAUDE.md auto-loading eliminates that line item entirely. As licenses grow, the savings scale linearly.

Research findings: what did not work in other rollouts

These are the recurring failure modes from large-scale Claude rollouts (NEC at 30,000 employees with vertical-specific training, Zapier with 800-plus internal Claude agents, the systemprompt.io 50-developer playbook). Every one of these has a named mitigation in the RS&H design.

Failure modeWhat it looks like in practiceRS&H mitigation
Over-restriction Users have to approve every file read. Tool gets abandoned within a day. Default to acceptEdits mode. Conservative permissions on writes only.
No training Users do not know what skills exist. Revert to raw chat. Cancel licenses. One-hour hands-on onboarding per cohort. Pilot members pair with new users for week one.
Missing CLAUDE.md standards Claude behaves inconsistently across teams. Users lose trust. Shared org CLAUDE.md as parent. Domain CLAUDE.md extends, never replaces.
No feedback loop Pilot complaints get ignored. Adoption stalls. Weekly 15-minute pilot retro for the first four weeks.
Skipped phases Rush from pilot to org-wide. Support structure is not there. Quality collapses. Hard gating metric at each phase exit (3+ sessions per week, 70%+ weekly active, etc.).
No named owner post-launch Managed settings file goes stale 90 days after rollout. No one notices. Working Group is the named owner. Quarterly review on the calendar.
Org content in personal CLAUDE.md Users put RS&H context in ~/.claude/CLAUDE.md. Drifts, leaks, no version control. Onboarding rule: personal CLAUDE.md is preferences only. Periodic survey to spot drift.

The single line. Orgs that fail rush past pilot before the one-week pairing discipline takes hold. Orgs that succeed treat the framework as infrastructure, not a tool, and name an owner.

Where this strains internally

  • Plugin release cadence vs. urgent fixes. Domain Stewards need a documented hotfix path.
  • Memory curation drift. Monthly distillation must be calendared and owned by name.
  • Skill sprawl. Quarterly audit. Fewer than three users on a local skill means deprecate or merge.
  • Permission drift. Quarterly justify-or-rollback review of any local settings overrides.
  • OneDrive sync conflicts. Multiple editors hitting the same context file. Mitigation: per-folder edit rights via SharePoint groups, only the Domain Steward edits the domain CLAUDE.md.
  • Two-CLAUDE.md problem. Users put org content in personal CLAUDE.md. Onboarding rule plus periodic survey catches it.

Cost risk: moving too fast on Fabric

The doubling concern. Phase 2 connects Claude to Microsoft Fabric. Today, a Power BI question costs Fabric capacity units. With Claude in the loop, the same question costs Fabric capacity units plus Anthropic tokens for the prompt, response, and any back-and-forth refinement. A user who runs one DAX query today might run six exploratory queries through Claude. Two meters spinning where one used to.

Where it gets worse fast

  • Direct Lake column scans on wide fact tables. Capacity unit consumption scales with columns hit, and Claude may select more than a human would.
  • XMLA queries that materialize full result sets when a TOPN would suffice.
  • Long sessions that re-query the same data because earlier results fell out of context.
  • Default model set to Opus instead of Sonnet (roughly 5x token cost for usually no quality gain in domain work).

Regulated and federal data

Claude Code is not HIPAA-covered today. Only the Claude chat product is covered under Anthropic's HIPAA-ready Enterprise plan. RS&H does federal and FDOT engineering work. Any data governed by an FDOT Data Use Agreement, CUI, ITAR, or FedRAMP requirements is out of scope for Claude Code prompts and memory until the Working Group has a separate compliance conversation with Anthropic Sales.

Security baseline

The single most important setting at enterprise scale is disableBypassPermissionsMode: true, deployed via Intune as a managed setting. Without it, any user can run --dangerously-skip-permissions and bypass every guardrail. Anthropic ships official Intune templates (PowerShell or ADMX, the standard Group Policy template format). RS&H should use the ADMX route. This needs IT ownership identified before Phase 1 build.

Phase 2: Microsoft Fabric integration

RS&H's Power BI reports already sit in Microsoft Fabric. Microsoft has open-sourced microsoft/skills-for-fabric, a collection of Claude-compatible skills that let agents read from and (when authorized) write to Fabric workloads. In Phase 2, Claude can directly query the Power BI semantic models, lakehouses, and warehouses RS&H already runs.

What it gives RS&H

  • Read access first. Claude answers questions like "what does the FDOT M-I-4 dashboard say about Q1 utilization?" by querying the semantic model. No manual data export.
  • Authoring access later. Under explicit human approval, Claude creates new semantic models, builds Spark ETL jobs, deploys reports. Hooked behind an approval gate.
  • Per-user auth. Users authenticate as themselves through Entra ID. Their existing Fabric permissions apply. Claude inherits.

Recommended sequence

  1. Pilot Fabric skills with the Tech domain only.
  2. Read-only first. Install consumption skills only.
  3. Per-domain auth boundaries already exist in Fabric. No new permission system needed.
  4. Authoring skills require explicit per-use approval through a hook.
  5. Roll to other domains after Tech proves the pattern. Finance is the natural second adopter.
